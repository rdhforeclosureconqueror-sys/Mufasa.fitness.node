F-004 ROUTE DISPOSITIONS:
POST /command: authenticated deprecated compatibility route, explicitly controlled by LEGACY_FALLBACK_ENABLED and action enforcement, rate limited, cross-user selection rejected, unknown commands rejected, and deprecation headers retained. A test-only createApp option preserves historical unit fixtures and is never enabled by production startup.
POST /api/speak: authenticated member route, JSON-only, allowlisted fields, 1,000-character text limit, provider-safe errors, and cost-bearing rate limit.
POST /api/challenges/pushup/results: authenticated member route with server-derived user identity, bounded integer repetitions, required submission identifier, and per-user replay rejection. Public leaderboard reads remain public and omit private contact/identity fields.
POST /api/pilot/events: authenticated telemetry route with an event-name allowlist, flat scalar payload allowlist, bounded strings, rejection of identity/prototype/nested fields, server-derived identity, and a rate limit.
F-004 status: CLOSED for the audited four routes.

RATE LIMITS:
A small server-authoritative in-memory fixed-window limiter protects speech, legacy command, push-up result, and pilot-event writes. It returns stable 429 JSON, preserves requestId, and emits Retry-After. Tests prove the first legitimate request proceeds and the over-limit request is rejected. This is single-process protection only; counters do not coordinate across replicas and reset on restart.

F-005 WORKOUT COMPLETION:
CLOSED. Generated completion now validates the persisted plan/session reference, completes the execution, and appends or confirms canonical workoutTracking within one userStore.updateUser callback and one persisted user write. The idempotency key is derived from executionId. Retry returns the completed execution without duplicate history; a formerly completed execution missing tracking is reconciled. Tests cover initial completion, repeated completion, one-write behavior, missing-tracking repair, plan immutability, foreign ownership, completed-mutation rejection, assigned-program precedence, and persisted progression compatibility. The synchronous repository update means same-process calls serialize at the JavaScript execution boundary; multi-process file-write serialization remains outside this persistence abstraction.

PARTIALLY CLOSED. config/route-authorization-contract.js inventories all 109 statically declared Express routes with method, path, authentication, roles, permissions, membership, ownership, feature flag, public-output, rate-limit, public-write, and compatibility policy. The audit test compares both directions and fails for missing/orphan routes, missing ownership/privilege, unclassified public writes, or missing output policy. Focused black-box tests cover anonymous denial on all four remediated writes, authenticated owner telemetry, body identity rejection, and malformed/unknown telemetry. Existing authentication/control-plane suites cover expired tokens, revocation metadata, permission denial, role allowlists, and protected member routes. A complete Phase 1–10 route-by-route cross-user/role/permission/membership/feature matrix remains required before full closure.

F-028 OUTPUT SANITIZATION:
PARTIALLY CLOSED. A shared safe-rendering utility provides textContent display and rejects javascript: and unsafe data: URLs without HTML double escaping. Tests cover tags, event attributes, quotes, ampersands, Unicode, and active URLs. A repository sink-inventory test scans browser JS/HTML and requires every dangerous sink file to have a reviewed justification. Confirmed generated-workout/Journey/nutrition templates retain their existing escaping contracts. Broad replacement of reviewed structural innerHTML templates is intentionally deferred; each remaining allowlisted sink still requires route-to-sink reachability validation before full closure.

ERROR SAFETY:
Touched authenticated routes use the existing central envelope for authentication/validation errors. Rate-limit responses include requestId. Speech provider failures no longer return raw provider bodies and never return secrets, stacks, or filesystem paths. Statuses used are 400 validation, 401 unauthenticated, 409 replay/reference conflict, 415 unsupported media type, 429 rate limit, 500 missing server configuration, and 502 provider failure.

COMPATIBILITY:
Old /command behavior allowed unauthenticated mutation when enabled and stored unknown actions. New behavior always authenticates production calls, derives/compares identity, and rejects unknown actions. LEGACY_FALLBACK_ENABLED=false remains the fail-safe rollback switch; action-level enforcement can block individual compatibility commands. Existing session clients retain deprecation headers and authenticated compatibility behavior. Speech and challenge browser consumers must now send the normal bearer token; challenge writes must include a stable submissionId. Public challenge reads are unchanged. allowInsecureTestRoutes exists only as an explicit createApp test-fixture option and is not read from environment or production startup.

ROLLBACK:

Rate limiter: reverting removes single-process protection and does not affect persisted data.
Generated completion: reverting can reintroduce split-write divergence. New workoutTracking records add idempotencyKey/source/trackedAt fields that legacy readers tolerate as additive JSON; rollback does not require migration, but a reverted completion may fail to recognize these keys and can append duplicate history.
Authorization contract: contract/test removal has no runtime data effect but removes route-addition assurance.
Output safety: reverting utility/client logging changes has no data effect but re-exposes unsafe URL and sensitive speech diagnostics. The sink allowlist is test-only documentation.
Challenge identity/replay: persisted challenge records now include additive userId and submissionId. Legacy readers ignore these fields; rollback is data-compatible but removes ownership/replay enforcement.
